1. **Менять юридический смысл** — формулировки обязательств, прав субъекта ПД, срока хранения, правовых оснований обработки, ссылок на законы (152-ФЗ, 149-ФЗ, ст. 18.1, ст. 19 и т.д.)
2. **Удалять разделы** из Политики или Согласия
3. **Расширять перечень обрабатываемых ПД** без одновременного обновления формы на лендинге и бэкенда (`api/lead.js`). Перечень в документах и в реальности **обязаны совпадать** — расхождение это нарушение ФЗ-152
4. **Публиковать биометрию, специальные категории ПД, финансовые данные** — документы под это не написаны
5. **Менять реквизиты ИП** без подтверждения от Наиля — ЕГРИП всегда первоисточник
6. **Делать чекбокс согласия pre-checked** — по ФЗ-152 и GDPR согласие должно быть активным действием
7. **Убирать серверную валидацию `body.consent === true` в `api/lead.js`** — клиентскую проверку обойти тривиально, сервер это единственный источник правды для юр-аудита

## Версионирование согласия (критично)

В `main.js` в обработчике submit формы есть строка (актуальная версия — `2025-10-15`):
```js
data.consent_text_version = '2025-10-15';
```

**Эта версия записывается в Airtable** (поле `Consent text version`) для каждого лида — чтобы можно было доказать, под какой редакцией документа человек дал согласие.

**Когда менять версию:**
- Любая правка **смыслового текста** Политики или Согласия → бампнуть дату на дату правки (формат `YYYY-MM-DD`)
- Чисто косметические правки (вёрстка, опечатки вне юр-формулировок) → можно не бампать
- Добавление нового поля в форму → **обязательно** бампать

**Где синхронизировать версию:**
1. `main.js` — `data.consent_text_version = 'YYYY-MM-DD'`
2. `legal/privacy-policy.html` — `<p class="legal-meta">Действует с DD.MM.YYYY</p>`
3. `legal/consent.html` — `<p class="legal-meta">Действует с DD.MM.YYYY</p>`
4. Не забудь обновить cache buster `?v=` в `index.html` после правок `main.js`

Старые версии документов для аудита хранятся в git history — отдельного архива не делаем.

## Чекбокс согласия в форме

Лежит в `index.html` внутри `#leadForm` перед кнопкой `.lead-submit`. Структура:

```html
<label class="form-consent">
    <input type="checkbox" name="consent" id="leadConsent">
    <span class="consent-box" aria-hidden="true"></span>
    <span class="consent-text">
        Нажимая «Отправить», я соглашаюсь с
        <a href="legal/privacy-policy" target="_blank" rel="noopener">политикой обработки персональных данных</a>
        и даю
        <a href="legal/consent" target="_blank" rel="noopener">согласие на обработку персональных данных</a>.
    </span>
</label>
```

**Правила:**
- `checked` атрибут **запрещён** (согласие = активное действие)
- `required` атрибут **не ставить** — на форме `novalidate`, валидация живёт в `main.js`
- Ссылки **всегда** `target="_blank" rel="noopener"` и **без** `.html` в href
- Визуал: стили в `style.css` в секции `/* ===== Consent checkbox ===== */`. Квадратный 20×20, чёрная рамка 2px, при отметке — оранжевый фон с белой галочкой через `::after`

## Бэкенд — `api/lead.js`

Серверная валидация согласия — **не удалять**:
```js
if (body.consent !== true) {
  return res.status(400).json({ error: 'Требуется согласие на обработку персональных данных' });
}
```

Проверяется **строго `=== true`**, не truthy. Причина: клиент может прислать `"on"`, `"true"`, `1` — всё это должно отвергаться, чтобы исключить случайную отправку без явного согласия.

Бэкенд записывает в Airtable таблицу `Leads` (base `appwANdQ0Txe6BRsy`) **5 специальных полей** для юр-аудита:

| Поле Airtable | Тип | Источник |
|---|---|---|
| `Consent given` | checkbox | всегда `true` при успешной валидации |
| `Consent timestamp` | dateTime (ISO, Europe/Moscow) | `body.consent_timestamp` — клиентское время отправки формы |
| `Consent text version` | singleLineText | `body.consent_text_version` — версия из `main.js` |
| `IP` | singleLineText | `req.headers['x-forwarded-for']` (первый IP в цепочке) |
| `User agent` | multilineText | `req.headers['user-agent']` |

Эти поля созданы в Airtable через MCP `mcp__Airtable__create_field` и **не должны удаляться**. Если переименовываешь поле в Airtable — синхронизируй ключ в `api/lead.js` одновременно, иначе запись в Airtable начнёт 422-ить.

Telegram-уведомление содержит строку `✅ Согласие на ОПД получено (v...)` — это удобно для визуального контроля, но юр-источник правды это Airtable.

## Cookies и Яндекс Метрика

На лендинге есть cookie-баннер + gate на загрузку аналитики. Это ортогональная система к form-consent (галочке в модалке) — у них **разные** согласия и **разные** механизмы хранения.

### Архитектура

- **Баннер** — `#cookieBanner` в `index.html` внизу файла. Изначально `hidden`. Показывается из JS если нет сохранённого решения. Две кнопки одного размера/контраста: «Только необходимые» и «Принять». По ФЗ-152 + ePrivacy.
- **Решение пользователя** хранится в `localStorage` под ключом `forge_cookie_consent`:
  ```json
  { "analytics": true, "version": "2025-10-15", "timestamp": "2025-10-15T09:12:44.123Z" }
  ```
- **Gate в `main.js`** — на DOMContentLoaded проверяет localStorage. Если решение есть и `analytics === true` — динамически подгружает Метрику. Если нет решения или версия policy бампнута — показывает баннер. Если есть решение с `analytics === false` — ничего не загружает.
- **Кнопка «Управление cookies»** в футере (`#manageCookies`) снова показывает баннер — так пользователь может отозвать или дать согласие позднее.
- **Метрика подгружается ДИНАМИЧЕСКИ**, не через тег в `<head>`. Критично: до согласия **никаких** запросов на `mc.yandex.ru` не уходит.
- **Webvisor включён с маскированием полей формы** — на `#leadForm` и всех её inputs стоит класс `ym-hide-content`, Метрика заменяет их содержимое на звёздочки. Имя, компания, контакт, задача в запись вебвизора НЕ попадают.

### Счётчик Метрики

В `main.js` есть константа (актуальное значение — счётчик `108474355`, зарегистрирован в кабинете `nmardamshin` на metrika.yandex.ru):
```js
const METRICA_COUNTER_ID = 108474355;
```

Если нужно сменить счётчик или добавить второй — замени это значение на новое числовое (без кавычек) и обнови cache buster `?v=` в `index.html` после правки. Паттерн загрузки таков:
- Согласившиеся пользователи грузят Метрику через dynamic script injection
- Отказавшиеся не грузят, пока не нажмут «Управление cookies» и не пересогласятся
- Если значение временно сделать `null` — gate не упадёт, баннер продолжит работать, просто Метрика не подгружается (no-op)

Не забудь обновить cache buster `?v=` в `index.html` после любой правки `main.js`.

### Версионирование cookie-согласия

Отдельная от form-consent переменная — `CONSENT_VERSION` в `main.js`:
```js
const CONSENT_VERSION = '2025-10-15';
```

Бампай эту версию **каждый раз**, когда в Политике меняется раздел 9 «Использование файлов cookies и метрических сервисов». Механика: если сохранённая версия в localStorage не совпадает с текущей `CONSENT_VERSION` — gate считает решение невалидным и показывает баннер заново. Так пользователи получают возможность переподтвердить согласие с новой редакцией.

**Что НЕ требует бампа `CONSENT_VERSION`:**
- Правки вёрстки/типографики баннера
- Косметика раздела 9 (исправление опечаток, перестановка абзацев)

**Что требует бампа:**
- Новый метрический сервис, кроме Яндекс Метрики
- Изменение набора собираемых данных
- Смена счётчика Метрики (спорно — юридически не обязательно, но если параноишь — бампай)
- Добавление любой категории cookies (рекламные, функциональные с трекингом и т.д.)

### Что НЕ подключать без перепроектирования баннера

Текущий баннер — **simple** (одна кнопка «Принять», одна «Отклонить»). Он закрывает только одну категорию cookies — аналитика. Если решишь добавить:
- **Рекламные пиксели** (Facebook Pixel, Google Ads, Яндекс Аудитории) → нужен **granular** баннер с отдельными тогглами. Текущую simple-реализацию нельзя использовать — по GDPR и ФЗ-152 нельзя объединять согласие на разные цели обработки в одну кнопку
- **Функциональные cookies с идентификатором** (например, запоминание корзины) → отдельная категория, тоже требует отдельного тоггла
- **Третьестороние встраивания** (YouTube embed, карты Google/Yandex) → нужен либо granular-баннер, либо отложенная загрузка с кликом пользователя

Если соберёшься расширяться — переделываем в `granular`-режим с модалкой настроек.

### Не трогать без юридической консультации

1. Текст **раздела 9** в `privacy-policy.html` — формулировки «передача обезличенного IP», «маскирование полей формы», ссылки на Политику Яндекса
2. Принцип «**ничего не грузится до согласия**» — если добавить preload/preconnect на `mc.yandex.ru` в `<head>`, это уже нарушение (обращение к третьему серверу до согласия)
3. **Равенство размера и контраста кнопок** «Принять» и «Отклонить» — под GDPR тёмные паттерны (большая оранжевая «Принять» + маленькая серая «Отклонить») считаются невалидным согласием
4. **Отсутствие pre-checked** — тот же принцип

## Куда копать если что-то сломалось

- **Форма отправляет без галочки** → баг в `main.js`, проверь порядок: валидация consent **до** `submitBtn.disabled = true`
- **Airtable 422 `UNKNOWN_FIELD_NAME`** → переименовали поле в Airtable, в `api/lead.js` осталось старое имя. Ключи в payload должны **совпадать символ-в-символ** с именами полей (включая пробелы: `'Consent given'`, а не `Consent_given`)
- **В Airtable пустое `Consent timestamp`** → клиент не прислал `consent_timestamp`, сервер fallback-нул на свой `new Date().toISOString()` — это штатно, но если массово, значит `main.js` сломан
- **Полиси страница не открывается по `/legal/privacy-policy`** → проверь `vercel.json` что `cleanUrls: true` остался, проверь что файл реально `legal/privacy-policy.html` (не `.htm`, не вложен глубже)
- **Браузер показывает старую версию** → cache buster не обновили
- **Cookie-баннер не показывается у нового пользователя** → проверь в DevTools → Application → Local Storage: есть ли ключ `forge_cookie_consent`? Если есть — это старое решение, удали и обнови страницу. Если нет — смотри консоль на ошибки инициализации (возможно сломан парсинг)
- **Баннер показывается каждый раз при перезагрузке** → localStorage не сохраняется (приватный режим браузера, заблокирован сторонним ПО), или версия `CONSENT_VERSION` не совпадает с сохранённой
- **Метрика не грузится после Accept** → проверь что `METRICA_COUNTER_ID` в `main.js` не `null`. Проверь в Network что запрос на `mc.yandex.ru/metrika/tag.js` реально уходит. Проверь в консоли что `window.ym` объект существует
- **Метрика грузится БЕЗ согласия** → критичный баг. Проверь что нет `<script>` тега Метрики прямо в `index.html` (должно быть только в JS, после Accept), нет `<link rel="preconnect" href="mc.yandex.ru">` в head. Проверь, что gate не вызывает `loadMetrica()` до проверки `getConsent()`
- **В Webvisor записываются данные форм** → класс `ym-hide-content` не применён к inputs формы или к самой форме. Проверь HTML вокруг `#leadForm`
